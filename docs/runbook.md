## 1. Performance Alerts

### 🔴 High CPU — `priority: critical`

**Trigger:** CPU ≥ 90% sustained for more than 2 minutes.

**Steps:**

1. Identify the consuming process:
   ```sql
   -- View open CPU tickets
   SELECT id, title, created_at, assignee
   FROM tickets
   JOIN ticket_categories  ON ticket_categories.id  = tickets.category_id
   JOIN ticket_statuses    ON ticket_statuses.id     = tickets.status_id
   WHERE ticket_categories.name = 'performance'
     AND ticket_statuses.name IN ('open', 'escalated')
   ORDER BY created_at DESC
   ```
2. Connect to the affected server and run `top` / Task Manager.
3. If it is a known process (backup, antivirus): wait 10 minutes and monitor.
4. If the process is unknown: escalate to the security team immediately.
5. Resolve the ticket with details of the identified process.

**SLA:** Response within 15 minutes. Resolution within 1 hour.

---

### 🟠 Disk Full — `priority: high`

**Trigger:** Disk usage ≥ 85%.

**Steps:**

1. Confirm the affected server and partition from the ticket.
2. Identify large files:
   ```bash
   # Linux
   du -sh /* 2>/dev/null | sort -rh | head -20

   # Windows PowerShell
   Get-ChildItem C:\ -Recurse -ErrorAction SilentlyContinue |
     Sort-Object Length -Descending | Select-Object -First 20 FullName, Length
   ```
3. Clean up safe candidates: old logs, temp files, memory dumps.
4. If nothing can be deleted: initiate a disk expansion request.
5. Critical threshold: ≥ 95% → escalate immediately.

**SLA:** Response within 1 hour. Resolution within 4 hours.

---

### 🟡 High RAM — `priority: high`

**Trigger:** RAM usage ≥ 90%.

**Steps:**

1. Check the trend — is it a spike or sustained usage?
2. Identify the consuming process.
3. If it is a known application with a memory leak: restart the service during a low-impact window.
4. Document in the ticket if the issue is recurring (candidate for capacity change).

**SLA:** Response within 1 hour. Resolution within 4 hours.

## 2. Network Alerts

### 🔴 HTTP Endpoint Down — `priority: high / critical`

**Trigger:** Health monitor receives HTTP status ≥ 400 or a timeout.

**Steps:**

1. Verify manually from another machine:
   ```bash
   curl -I https://affected-url.com
   ```
2. Check if the server is up (ping, RDP).
3. Review web server logs (IIS, nginx, Apache).
4. If the server responds but the service does not: restart the service.
5. If the server does not respond: escalate to infrastructure.

**Use `priority: critical`** if the endpoint is production-facing or affects external users.

**SLA critical:** Response 15 min. Resolution 1 hour.

---

### 🟠 VPN — Multiple Users Affected

**Trigger:** 3 or more VPN reports within 30 minutes.

**Steps:**

1. Determine scope: does it affect everyone or a specific region/office?
2. Review VPN server logs.
3. Check for recent changes (updates, firewall rule changes).
4. Communicate status to affected users while investigating.
5. If not resolved within 30 minutes: escalate to networking.

**Note:** Create a single group ticket instead of one per user.

## 3. Security Alerts

### 🔴 SSL Certificate Expiring — `priority: high`

**Trigger:** Certificate expires in ≤ 14 days.

**Steps:**

1. Confirm the domain and expiry date:
   ```bash
   echo | openssl s_client -connect domain.com:443 2>/dev/null |
     openssl x509 -noout -dates
   ```
2. Identify the certificate owner (IT, DevOps, or vendor).
3. Start the renewal process with the responsible party.
4. Document the committed renewal date in the ticket.

**Note:** If the cert has already expired → `priority: critical`, escalate immediately.

---

### 🔴 Unknown Process with High Resource Usage

**Trigger:** High CPU/RAM with an unidentified process.

**Steps:**

1. **Do not restart the server** — it may destroy evidence.
2. Capture process information:
   ```powershell
   Get-Process | Sort-Object CPU -Descending | Select-Object -First 10
   ```
3. Escalate to the security team immediately with the captured process list.
4. The ticket automatically moves to `priority: critical`.

## 4. Hardware Alerts

### 🟠 Printer Offline

**Steps:**

1. Check physical status (paper, toner, error messages on display).
2. Restart the print spooler:
   ```powershell
   Restart-Service Spooler
   ```
3. If still offline: check the printer's network connectivity.
4. If it is a physical hardware problem: create a field support ticket.

## 5. Software & Access Alerts

### 🟡 Locked Account

**Steps:**

1. Verify the user's identity — never unlock without verification.
2. Unlock the account in Active Directory:
   ```powershell
   Unlock-ADAccount -Identity "username"
   ```
3. Reset the password if the user requests it.
4. Document: how many failed attempts? From which IP?

---

### 🟡 Onboarding — New Account

**Checklist:**

- [ ] Create account in Active Directory
- [ ] Assign groups based on department
- [ ] Create mailbox
- [ ] Assign licenses (M365, etc.)
- [ ] Configure MFA
- [ ] Send credentials through a secure channel
- [ ] Confirm access with the user on their start date

## 6. Escalation & SLA

### SLA Table

| Priority | Response | Resolution | Auto-escalation |
|----------|----------|------------|-----------------|
| Critical | 15 min   | 1 hour     | At 15 min with no response |
| High     | 1 hour   | 4 hours    | At 1 hour with no response |
| Medium   | 4 hours  | 24 hours   | At 4 hours with no response |
| Low      | 24 hours | 72 hours   | At 24 hours with no response |

### When to Escalate Manually

- The issue affects more than 10 users simultaneously.
- There is suspicion of a security incident.
- The issue involves customer data or sensitive information.
- You do not have the access or permissions needed to resolve it.
- The affected system is business-critical (ERP, CRM, production).

### Escalation Matrix

| Issue Type | Team |
|------------|------|
| Network / firewall / VPN | Networking |
| Servers / infrastructure | Sysadmin / DevOps |
| Security / malware | Security team |
| Business applications | Development / vendor |
| Physical hardware | Field support |

## 7. Ticket Closure Procedure

Before resolving a ticket, verify:

- [ ] The issue was confirmed as resolved by the user or the monitor.
- [ ] The root cause was documented in the ticket description.
- [ ] The applied solution was documented.
- [ ] If it is a recurring issue: a follow-up or improvement ticket was created.
- [ ] The `resolved_at` timestamp was recorded correctly in the system.

```sql
-- Find resolved tickets missing root cause documentation
SELECT id, title, resolved_at
FROM tickets
WHERE status_id = (SELECT id FROM ticket_statuses WHERE name = 'resolved')
  AND (description IS NULL OR LEN(description) < 20)
ORDER BY resolved_at DESC
```
